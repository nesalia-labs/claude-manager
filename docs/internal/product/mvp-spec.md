# MVP Specification — claude-manager v1

> Last updated 2026-06-29 after v0.4 perf + UI upgrades.

## Summary

A Windows-only TUI that lists every Claude Code session on the local machine, grouped by project, with drill-down into subagents and tool activity. Read-only. Local. Updated live. Distributed via `bunx`.

## Reference implementation

Our v1 scope is modeled on two reference projects:

- **[cctop (stefanprodan)](https://github.com/stefanprodan/cctop)** — Bun + TS top-style monitor. Primary reference for v1 feature set, keyboard model, and architecture instincts. macOS/Linux only; we are the Windows port. **We depart from cctop on runtime:** cctop is Bun-only; we are Node 22 + npm.
- **[cctop (DeanLa)](https://github.com/DeanLa/cctop)** — Python htop-style. Secondary reference for the granular status taxonomy and rich column model. Some of its status labels (`thinking`, `awaiting plan`, `needs input`) inform our v2 status expansion.

Feature parity at v1 with cctop (stefanprodan):

| Feature | cctop | claude-manager v1 |
| --- | --- | --- |
| All sessions in one view | ✅ | ✅ (project-grouped variant) |
| Sub-agent tree | ✅ | ✅ (metadata only, drill in v2) |
| Per-session detail view | ✅ | ✅ |
| Live TUI navigation | ✅ | ✅ |
| Filter on the fly | ✅ | ✅ |
| Quit a session (`x`) | ✅ | ❌ — strict read-only |
| Reclaim orphan ports (`f`) | ✅ | ❌ — post-v1 |
| Sub-process / TCP port tree | ✅ | ❌ — post-v1 |
| Session history dashboard (`h`) | ✅ | ❌ — post-v1 |
| Usage limits (5h/7d) | ✅ (opt-in hook) | ❌ — post-v1 |
| `--once` / `--json` for scripting | ✅ | ❌ — post-v1 |
| **Windows** | ❌ | ✅ |
| **Runtime** | Bun-only | **Node 22 + npm (we DEPART)** |

## In scope

### F1 — Project grouping

The top-level layout is a **vertical list of projects**, each acting as a collapsible section containing its sessions.

For each project, we show:

| Field | Source | Required |
| --- | --- | --- |
| Project name | registry `cwd` field for any session in the project (canonical) | Yes |
| Project path | tilde-replaced registry `cwd` (e.g. `~/code/myapp`) | Yes |
| Session count | sessions in this project (live + recently done) | Yes |
| Active sessions | count of sessions with status `running` | Yes |

#### Project grouping rules

- One project per distinct `cwd` value seen across the registry (`~/.claude/sessions/<pid>.json`).
- Project sections are sorted alphabetically by project path.
- A project is shown if it has **at least one session modified in the last 7 days**. Older projects are filtered out. (Tunable post-v1.)

#### Encoding notes

The `~/.claude/projects/<encoded>/` directory name is generated by `cwd.replace(/[^a-zA-Z0-9]/g, "-")` — every non-alphanumeric (including `:`, `\`, `-`, `.`, `/`) maps to the same `-`. **We do NOT rely on decoding this for display.** Instead, we use the registry's `cwd` field as the canonical project label (it's the full original path). This sidesteps the lossy encoding entirely.

### F2 — Sessions list (per project)

Inside each project section, one row per session. Auto-refreshing.

**Session discovery (canonical path):** each row comes from a `~/.claude/sessions/<pid>.json` file. The PID is matched against the live process table.

**Fallback path:** if a Claude-looking process exists with no registry entry, we discover its transcript by listing `~/.claude/projects/<encoded-cwd>/` and picking the most recently modified JSONL whose mtime is ≥ process start − 60s.

Each row shows:

| Field | Source | Required |
| --- | --- | --- |
| Session name | registry `name`, or summary of first user prompt | Yes |
| Status | derived (see Status Derivation below) | Yes |
| Model | most recent `assistant.message.model` field (parsed from transcript tail) | Yes |
| Context tokens | sum of input + cache_read + cache_creation from latest assistant usage | Yes |
| Duration | `now - registry.startedAt` | Yes |
| Last activity | `now - last_entry_timestamp`, relative time | Yes |
| Subagent count | live subagents under `<id>/subagents/agent-*.jsonl` | Yes |
| Tool call count | count of tool_use blocks in main transcript (cached) | Yes |

#### Status values

The status taxonomy is **three values**, deliberately simpler than competitor suites:

- **`running`** — (1) registry `status === "busy"` **OR** (2) `now - last_transcript_mtime < N1`.
- **`idle`** — process is alive AND `now - last_transcript_mtime >= N1` AND `< N2`.
- **`done`** — no live process for this PID **OR** `now - last_transcript_mtime >= N2`.

Thresholds (cctop-style constants, tunable in v1.x):
- `N1 = 10s` (running → idle boundary; "CC has gone silent recently")
- `N2 = 1h` (idle → done boundary; but we display `done` rows briefly then auto-hide)

#### Status derivation

```ts
function deriveStatus(row: SessionRow, nowMs: number): Status {
  const alive = row.processSnapshot != null;
  const lastMs = Math.max(row.registryUpdatedAt ?? 0, row.transcriptMtimeMs);
  const age = nowMs - lastMs;
  if (!alive || age >= N2) return "done";
  if (row.registryStatus === "busy" || age < N1) return "running";
  return "idle";
}
```

PID reuse guard: if registry `startedAt` doesn't match process `startSec` (±60s) or is in the future, drop the registry entry (PID was recycled).

### F3 — Session detail

Selecting a session opens a detail view with two panes side-by-side (or stacked vertically on narrow terminals):

#### F3.1 — Subagents pane

Lists every **live** subagent spawned by this session. Live means: file `agent-<uuid>.jsonl` under `<session-id>/subagents/` whose mtime is within `SUBAGENT_BUSY_MS` (3 minutes) AND either was modified within `SUBAGENT_LIVE_MS` (20s) OR its last entry indicates an in-flight tool call (assistant `tool_use` block pending, or user `tool_result` block just arrived).

Each row:

| Field | Source |
| --- | --- |
| Activity | the agent's most recent `tool_use` block, formatted via `describeAssistant()` ("Read: src/foo.ts", "Bash: npm test", "Explore: …") |
| Model | latest `assistant.message.model` from the agent transcript |
| Context tokens | `input + cache_read + cache_creation` from the agent's latest usage |
| Uptime | `now - birthtime` (falls back to mtime if no birthtime) |

The pane is sorted by context size, descending (biggest consumers first). Subagents whose transcripts are missing or unreadable are silently dropped — we do not show stale rows.

**Subagent directory claim** is sequential, not parallel: two sessions in the same project can fall back to the same transcript; only the first to walk its `<id>/subagents/` claims the agents. This mirrors cctop's `attachSubagentsInOrder` and avoids double-listing.

#### F3.2 — Tool log pane

Lists every `tool_use` block from the parent session's JSONL, in chronological order. Each row:

| Field | Source |
| --- | --- |
| Time | entry timestamp (HH:MM:SS) |
| Tool name | `tool_use.name` (`Read`, `Edit`, `Bash`, `Grep`, `Glob`, `Agent`, etc.) |
| Target | `describeAssistant(tool_use)` → "Read: src/foo.ts", "Bash: npm test", "Agent: <description>", etc. |
| Status | `pending` (tool_use emitted, no result yet), `done` (matching tool_result present), `error` (tool_result.is_error === true) |
| Duration | `tool_result.timestamp - tool_use.timestamp` if `done` |

The tool log is read-only — there is no expand-to-see-tool-input-or-result action in v1.

### F4 — Live updates

The TUI refreshes when:

1. A new JSONL file appears under `~/.claude/projects/<encoded>/`.
2. An existing JSONL file is appended to.
3. A CC process starts or exits.
4. The polling fallback interval (2s) elapses.

Implementation: `fs.watch` on `~/.claude/projects/` (recursive) + 2s polling. Windows uses `ReadDirectoryChangesW` under the hood.

Re-scan strategy: when a file event fires, re-parse only the affected file(s) (cheap JSONL append reads). The full tree is rebuilt only on boot or after 30s of inactivity.

### F5 — Distribution

- Published to npm as `@claude-manager/cli` (from `packages/claude-manager/` in the monorepo).
- Install: `npm install -g @claude-manager/cli`.
- Run: `claude-manager` from any directory.
- Node 22 LTS minimum. Engines requirement in `packages/claude-manager/package.json`: `"node": ">=22"`.
- Monorepo context: developed inside a Turborepo + pnpm workspaces repo, but the published artifact is the single npm package. `pnpm --filter @claude-manager/cli build` produces `dist/`; `pnpm publish --filter @claude-manager/cli` pushes to the registry.

### F6 — Keyboard model

| Key | Action |
| --- | --- |
| `↑` / `↓` or `j` / `k` | Move selection (across projects and sessions) |
| `←` / `→` or `h` / `l` | Collapse / expand project section |
| `Enter` | Open session detail |
| `Esc` | Back to project list / clear filter |
| `Tab` | Switch between subagents pane and tool log pane (in detail view) |
| `/` | Filter by project path / session name |
| `y` | Copy `claude --resume <id>` of focused session to clipboard |
| `r` | Force refresh |
| `?` | Show help overlay |
| `q` / `Ctrl+C` | Quit |

## Out of scope (explicitly)

The following are **not** in v1 and should not be built unless re-scoped:

- Token / cost analytics (charts, daily totals, model breakdown).
- Task board view (kanban).
- Sending prompts to / controlling sessions.
- Multi-user / team / cloud sync.
- Cross-platform support (Linux, macOS).
- Historical session browser (only "recently active" sessions show).
- Custom themes / color customization.
- Mouse support.
- Export of analytics to CSV/JSON.
- Codex CLI monitoring (open to other CLIs is post-v1).
- Drill-down into tool call inputs / outputs (the log shows names + targets, not full payloads).
- Drill-down into subagent sessions (we show metadata only, not the subagent's own transcript).

## Acceptance criteria

v1 is accepted when **all** of the following are true:

1. On a clean Windows 11 + Node 22 install, `npm install -g claude-manager && claude-manager` opens within 3 seconds (5 seconds on cold install).
2. With three concurrent Claude Code sessions in different projects, all three appear under the right project sections within 5 seconds of launch.
3. Killing a CC process transitions its row to `done` within 10 seconds (subprocess polling cadence is the lower bound).
4. Resuming a session from another terminal causes its row to appear with status `running`.
5. A session that spawns two `Agent` (Explore) subagents shows both in the subagents pane, with correct type and status.
6. A session that calls `Read`, `Edit`, `Bash`, and `Grep` shows all four in the tool log in the right order.
7. Pressing `y` on a session row puts a valid `claude --resume <id>` command on the clipboard.
8. The tool does not write to `~/.claude/` (verified by file mtime / hash check after a 5-minute run).
9. CPU usage is below 3% on an idle Windows machine with 0 active CC sessions. (Higher than cctop because subprocess polling adds a baseline cost.)
10. Memory usage is below 120 MB during a 30-minute run with 5 active sessions across 3 projects.
11. The tool works on Windows 10 (build 19045) and Windows 11.
12. Works on a Sentinel One-protected machine with no admin rights (no FFI to ntdll/iphlpapi; only signed-Microsoft subprocesses).

## Open questions for v1

- npm package name: `@claude-manager/cli` vs unscoped `claude-manager`?
- Encoding disambiguation: do we ship v1 with the `?` badge and defer the `cwd` cross-reference to v1.1, or block on disambiguation first?
- Detail pane layout: side-by-side vs vertical? Driven by terminal width (TBD threshold).
- "Recently active" filter: 7 days is a guess. Validate against real CC usage patterns.
- Should `done` sessions be filtered out entirely after some grace period (e.g. 5 minutes)?
